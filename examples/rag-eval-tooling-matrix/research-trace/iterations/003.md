## Findings

- **fact**: Evidently is a plausible primary matrix row for notebook/report-driven RAG quality checks: official docs demonstrate a local open-source workflow that evaluates retrieval context quality and generation quality, then produces dataframe and visual report outputs. (refs: https://docs.evidentlyai.com/examples/LLM_rag_evals)
- **fact**: W&B Weave is a plausible primary matrix row for teams that want managed observability plus evaluation: its official RAG tutorial tracks retrieval steps, evaluates responses with LLM judges, and uses datasets/scorers inside Weave evaluations. (refs: https://docs.wandb.ai/weave/tutorial-rag)
- **synthesis**: LlamaIndex evaluators should be treated as a framework-fit caveat row unless the production RAG stack already uses LlamaIndex. The official docs are strong on response/retrieval evaluation and generated test questions, but the capability is primarily framework-native rather than a standalone evaluation platform. (refs: https://developers.llamaindex.ai/python/framework/module_guides/evaluating/)
- **synthesis**: Haystack evaluation should be treated as a framework-fit caveat row unless the production RAG stack already uses Haystack. Official docs cover component and pipeline evaluation, RAG model-based evaluation, statistical retriever metrics, and Ragas/DeepEval integrations, but the strongest value is inside Haystack pipelines. (refs: https://docs.haystack.deepset.ai/docs/model-based-evaluation, https://docs.haystack.deepset.ai/docs/evaluation)
- **risk**: OpenAI Evals should not be scored as a long-term primary RAG evaluation stack because official OpenAI docs now mark the Evals platform as deprecated, with read-only status for existing users on 2026-10-31 and shutdown on 2026-11-30. It belongs in a caveat or legacy row unless the final matrix explicitly covers OpenAI-specific evaluation surfaces. (refs: https://developers.openai.com/api/docs/guides/evals)
- **plan**: The analysis phase should separate capability scoring from packaging/governance scoring. Public eval capability docs are enough to classify RAG metrics, CI/regression fit, observability, datasets, and framework fit, but license, self-hosting, SaaS, RBAC, retention, and enterprise controls need dedicated public-source verification. (refs: https://docs.evidentlyai.com/examples/LLM_rag_evals, https://docs.wandb.ai/weave/tutorial-rag, https://developers.llamaindex.ai/python/framework/module_guides/evaluating/, https://docs.haystack.deepset.ai/docs/model-based-evaluation, https://developers.openai.com/api/docs/guides/evals)
