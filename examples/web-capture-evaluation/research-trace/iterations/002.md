## Findings

- **synthesis**: The example should organize risks into five reader-facing categories: permission/politeness, request safety, URL and network containment, conversion/Markdown handling, and prompt-injection boundaries. (refs: https://datatracker.ietf.org/doc/html/rfc9309, https://datatracker.ietf.org/doc/html/rfc9110, https://docs.python.org/3/library/urllib.parse.html, https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html, https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html, https://spec.commonmark.org/0.31.2/)
- **synthesis**: A safe capture workflow can be presented as six stages: intake, preflight, fetch, convert, analyze, and package. The workflow should require URL policy checks before fetch, redirect revalidation, bounded content, provenance metadata, and untrusted-content handling during analysis. (refs: https://datatracker.ietf.org/doc/html/rfc9110, https://docs.python.org/3/library/urllib.parse.html, https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html, https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html)
- **plan**: The five requested files map cleanly to package roles: README.md for purpose and safety boundaries, prompt.md for the worker contract, research-plan.md for gates and stages, sources.md for official references, and final-report.md for the evaluated workflow and artifact checklist. (refs: https://datatracker.ietf.org/doc/html/rfc9309, https://datatracker.ietf.org/doc/html/rfc9110, https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html, https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html)
- **synthesis**: The public example should not include a live capture transcript in the initial five-file package. A transcript is useful only as a later synthetic or explicitly sanitized extension, because the requested deliverable is a package template and should avoid redistribution concerns. (refs: https://datatracker.ietf.org/doc/html/rfc9309, https://spec.commonmark.org/0.31.2/)
- **synthesis**: The example should stay converter-neutral. MarkItDown can be named as one illustrative converter with safety caveats, but the workflow should describe hardened fetch and bounded conversion independent of any required dependency. (refs: https://github.com/microsoft/markitdown, https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)
